## Problem Statement

Heavy AI contributors lose time re-finding old Codex or Claude conversations when a work item wakes back up days or weeks later.

Today the memory lives inside the AI client. That is the wrong place. Search is weak, previews reflect how a conversation started rather than what it solved, issue-id search is unreliable, histories get buried under many unrelated threads, and work often spans multiple local projects and machines.

The real product is not "a Drupal.org helper." The real product is a local-first memory layer that attaches prior AI conversations to the work object itself. For launch, the work object is a Drupal.org issue page. Later it can expand to Jira tickets, Slack threads, emails, and notifications that refer to the same underlying work item.

## What Makes This Cool

You land on an issue page and the right old AI conversation is just there.

That is the whole trick.

Instead of asking the contributor to remember which AI tool they used, which machine they used it on, which repo they were in, or what the first prompt looked like, the system reconciles all of that around the work object. It turns a buried chat history into durable working memory.

For heavy AI users, that changes issue work from "reconstruct context again" to "continue where I left off."

## Constraints

- Must stay local-first. No hosted team service in v1.
- Must target AI-heavy contributors first, not generic Drupal contributors.
- Must keep Beads as the portable, cross-machine issue-to-conversation index.
- Must treat Drupal.org issue pages as connector #1, not the final platform.
- Must not require users to adopt Scott's exact workstation layout.
- Must not ship unsafe local command execution defaults to other users.
- Must preserve the current "open work object, recover prior conversation fast" moment with as little setup as possible.

## Premises

1. This should launch as a personal/local power tool for heavy AI contributors, not a team knowledge platform.
2. Beads is not a random implementation detail. It is the portable issue-to-conversation index.
3. The entry point is the work object or notification, not the AI client.
4. Jira, Slack, and email are future connectors, not launch scope.

## Approaches Considered

### Approach A: Productize The Current Wedge

Ship a cleaned-up version of the current Drupal.org-specific tool. Browser extension, tiny local server, Beads-backed issue mapping, and Codex/Claude resume links.

- Effort: M
- Risk: Low
- Pros: Fastest path to users. Preserves the magic. Smallest packaging lift.
- Cons: Bakes Drupal.org assumptions into the product. Future Jira/Slack/email support likely requires refactoring core concepts later.

### Approach B: Connector Architecture From Day One

Turn the product into a general local memory router with pluggable work-object connectors and AI-conversation adapters. Drupal.org issues are connector #1. Codex and Claude are conversation adapters. Beads remains the shared index.

- Effort: L
- Risk: Medium
- Pros: Clean long-term model. Easier path to Jira/Slack/email later. Makes the product idea clearer than "local helper for one website."
- Cons: Higher risk of abstraction drift. Needs discipline to avoid building a framework before proving adoption.

### Approach C: Index-First Memory Daemon

Make the core product a local daemon that reconciles identifiers from issue URLs, ticket IDs, Slack references, email subjects, and AI session history into one local memory graph. Browser integrations become thin views over that daemon.

- Effort: M-L
- Risk: Medium-High
- Pros: Closest to the 10x vision. Makes notification-driven recovery possible.
- Cons: Harder to explain, package, and validate as v1. Delays the sharpest initial user value.

## Recommended Approach

Proceed with **Approach B: Connector Architecture From Day One**, but with an extremely narrow launch cut.

This is the important nuance: do not build a broad connector ecosystem before shipping. Build the connector contract first, then ship exactly one work-object connector and exactly two conversation adapters:

- Work-object connector: Drupal.org issue pages
- Conversation adapters: Codex and Claude
- Shared portable index: Beads

The architecture should make Jira/Slack/email obviously possible later, but the first release should still feel like one small sharp tool.

The product model:

- A **work object** is the thing the user is looking at, starting with a Drupal.org issue URL and issue id.
- A **memory source** is an AI conversation system, starting with Codex and Claude local history.
- An **index** links work objects to one or more conversation handles, with Beads as the user-editable portable source of truth.
- A **surface** is where recovery happens, starting with a browser extension on the issue page plus a tiny local companion.

The architecture succeeds if users understand none of those terms and still feel the outcome instantly.

## Open Questions

1. What is the stable connector contract for a work object? URL only, canonical id, or both?
2. What is the stable contract for a conversation adapter? Resume command, session id, preview text, timestamps, confidence score?
3. Should Beads remain the only writable index in v1, or should the companion also emit local cache metadata?
4. How should the tool handle one issue mapped to multiple conversations across tools and machines?
5. What is the secure default for launch actions? Copy-only by default, or opt-in local execution after explicit setup?
6. Is Chrome-extension-first enough for launch, or should there also be a small local CLI like `work-memory open <issue-url>`?
7. What exact install experience is acceptable for power users? Manual unpacked extension plus `npm start`, or a more polished packaged installer?

## Success Criteria

- A heavy AI contributor can install the first public version in under 10 minutes.
- When they open a supported Drupal.org issue, they see relevant prior Codex or Claude conversations without manual history digging.
- They can resume the correct conversation in one step or less than 10 seconds.
- Beads-backed mappings survive switching machines.
- The codebase exposes a clear connector boundary so Jira can be added later without rewriting the core model.
- Early users describe it as "this saves me from hunting through old chats" rather than "nice extension."

## Distribution Plan

Launch as a GitHub-hosted open source project with:

- A packaged browser extension build for manual install in Chromium-based browsers.
- A local companion distributed either as a small Node package (`npx` or global install) or as a single documented repo checkout if packaging is not ready yet.
- A config file or explicit environment variables for workspace root, Beads location, supported AI tools, and launch behavior.

CI/CD should cover:

- Automated tests for connector and adapter contracts
- Lint or syntax checks for extension and companion code
- Release artifacts for the extension bundle
- Versioned release notes that describe supported connectors and adapters

No cloud service. No account system. No telemetry dependency for core functionality.

## Next Steps

1. Define the connector model in plain English: work object, conversation adapter, index, surface.
2. Refactor the current Drupal.org implementation so Drupal.org becomes connector #1 instead of the whole app.
3. Refactor Codex history scanning and Claude support behind a shared adapter contract.
4. Move every hard-coded path and machine assumption into explicit config.
5. Make local launch actions safe by default, with copy-first behavior and opt-in execution.
6. Package the smallest usable install flow for one external power user.
7. Test with 2-3 heavy AI Drupal contributors on multiple machines before widening scope.

## Engineering Review Decisions

Generated by `/plan-eng-review` on 2026-04-09.

- Keep one repo with internal module boundaries. Do not split into multiple repos or public packages yet.
- Ship Drupal.org issues plus Jira-style `atlassian.net/browse/KEY-123` tickets as the first connector set.
- Split the data model into portable work anchors and machine-local session bindings.
- Use Beads for portable work anchors and aliases. Use a tiny local Direct Resume store for machine-local bindings, cache, and install token.
- Keep localhost HTTP as the v1 transport, hardened with loopback binding, extension-origin allowlist, and per-install token.
- Resolve APIs by full `WorkObject`, not bare issue strings.
- Make `Exec` a configurable local capability, off by default. `Copy` stays available for all candidates.
- Let the extension send only `{ url, page_title }`; connector matching and canonicalization live in the companion.
- Remove the Drupal dashboard from the shared public plan. It can remain prototype-only or be deleted.
- Keep Node's built-in test runner. Add companion HTTP integration tests and a small DOM/message harness instead of full browser E2E in v1.
- Add file-mtime-aware local adapter caches so normal page loads do not rescan large AI history files.
- Store `workspace_path` on local session bindings and session candidates.
- Add adapter liveness checks so stale local bindings are hidden or pruned.
- Let `/api/resolve` accept a `metadata` object with page-derived hints such as canonical URL and issue key.
- Weight full URL and connector-key evidence above bare numeric aliases.
- Add explicit setup/pairing for the extension-to-companion token.
- Require exec actions to launch a visible terminal/editor surface; never launch interactive CLI agents headlessly.
- Persist only explicit or user-confirmed local bindings. Keep inferred and suggested matches ephemeral or TTL-cached.
- Move ranking into core. Connectors and adapters emit typed evidence.
- Make `/api/resolve` return opaque candidate references, not raw resume commands.
- Make `/api/resume` re-run liveness and policy checks and return typed action objects.
- Add `protocol_version`, explicit schemas, structured error codes, install-generated UUID `machine_id`, and short-lived single-use pairing token rules.
- Add one seam-level smoke test crossing extension message handling, localhost auth, resolve, and resume with fake connector/adapter implementations.
