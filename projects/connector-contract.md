# Connector Contract

This document defines the smallest contract needed for `/plan-eng-review`.

The goal is not a platform. The goal is one reliable flow:

> From a work-object page in the browser, directly resume the right local coding-agent session.

## Core concepts

### Work object

A browser-visible object that represents a unit of work.

Examples:

- Drupal.org issue
- Jira ticket

Every work object must have:

- `kind`
- `canonical_id`
- `canonical_url`
- `display_title`
- `source`

Example:

```json
{
  "kind": "drupal.issue",
  "canonical_id": "drupal:canvas:3558241",
  "canonical_url": "https://www.drupal.org/project/canvas/issues/3558241",
  "display_title": "3558241 - Resolve MR !331 unresolved review comments",
  "source": "drupal"
}
```

Jira example:

```json
{
  "kind": "jira.issue",
  "canonical_id": "jira:acquia.atlassian.net:PROS-370",
  "canonical_url": "https://acquia.atlassian.net/browse/PROS-370",
  "display_title": "PROS-370",
  "source": "jira"
}
```

### Portable work anchor

A portable record that says this work object is tracked and can carry shared aliases or hints across machines.

It must not store machine-local session IDs as the source of truth.

Every portable work anchor must have:

- `work_object_id`
- `origin`
- `created_at`
- `last_seen_at`
- `aliases`
- `shared_summary`

Example:

```json
{
  "work_object_id": "drupal:canvas:3558241",
  "origin": "explicit_link",
  "created_at": "2026-04-08T22:00:00Z",
  "last_seen_at": "2026-04-08T22:10:00Z",
  "aliases": [
    "3558241",
    "https://www.drupal.org/project/canvas/issues/3558241"
  ],
  "shared_summary": "Reviewer follow-up on unresolved MR comments."
}
```

### Local session binding

A machine-local record that links a portable work object to a specific resumable local session.

Local session bindings are authoritative user-created state. Persist only explicit or user-confirmed links here.

Every local session binding must have:

- `work_object_id`
- `machine_id`
- `agent`
- `session_id`
- `workspace_path`
- `created_at`
- `last_seen_at`
- `last_verified_at`
- `state`

`machine_id` must be an install-generated UUID, not a hostname.

Allowed `state` values:

- `active`
- `stale`
- `hidden`
- `pruned`

Example:

```json
{
  "work_object_id": "drupal:canvas:3558241",
  "machine_id": "9c2f53f8-7e9d-4b16-a0cb-355f20d63e67",
  "agent": "codex",
  "session_id": "019c0cd9-4367-77d0-a0a2-a5ebbb15ab27",
  "workspace_path": "/Users/scott/dev/drupal-contrib/projects/canvas",
  "created_at": "2026-04-08T22:00:00Z",
  "last_seen_at": "2026-04-08T22:10:00Z",
  "last_verified_at": "2026-04-08T22:10:00Z",
  "state": "active"
}
```

### Session candidate

A resumable session surfaced by an agent adapter before final ranking.

Session candidates are heuristic discovery output. They are not authoritative bindings until the user explicitly links or confirms one.

Fields:

- `candidate_ref`
- `agent`
- `session_id`
- `resume_label`
- `workspace_path`
- `summary`
- `first_seen_at`
- `last_seen_at`
- `evidence`

`candidate_ref` is an opaque reference generated by the companion for the current resolve result. It is the only candidate identifier the extension should send back to `/api/link` or `/api/resume`.

Do not include raw shell commands in `SessionCandidate`.

### Evidence

Evidence is typed input for core ranking.

Fields:

- `type`
- `value`
- `weight`
- `source`

Initial evidence types:

- `explicit_binding`
- `canonical_url`
- `connector_key`
- `project_scoped_id`
- `bare_numeric_id`
- `repo_proximity`
- `recency`

## WorkObjectConnector contract

Each connector must implement:

### `match(url: string): boolean`

Returns whether the connector can handle this page.

### `canonicalize(input): WorkObject`

Accepts a browser URL and optional page metadata.
Returns a stable canonical work object.

Metadata is a hint, not trusted authority. Connectors may use page-provided data such as `og:url`, DOM issue keys, project labels, or ticket titles to improve canonicalization, but must still validate against the supported URL shape.

### `aliases(workObject): Evidence[]`

Returns typed evidence hints that may appear in local history.

Examples:

- full URL
- issue id
- short URL
- project issue path
- Jira issue key

### `display(workObject): object`

Returns human-facing text for the extension.

## SessionAdapter contract

Each agent adapter must implement:

### `discover(context): SessionCandidate[]`

Returns resumable local sessions relevant to the local machine.

## Ranking contract

Core owns ranking. Adapters emit normalized evidence, and core converts that evidence into ordered matches.

Ranking uses:

- explicit link evidence
- full canonical URL mention, strongest inferred signal
- connector-specific key mention, for example `PROS-370`
- project-prefixed Drupal issue evidence
- bare Drupal numeric issue id mention, weak signal only
- repo proximity
- recency

Bare numeric aliases must not outrank full URL, project, or explicit binding evidence.

### `isAlive(candidate): boolean`

Returns whether the underlying local session still exists and is resumable.

Adapters should use this during resolve/link flows so stale local bindings can be ignored or pruned before the UI presents a broken Resume action.

### `resumeAction(candidate, mode): ResumeAction`

Returns either:

- copy-safe action
- exec-safe action

No connector may emit shell commands directly. Only the session adapter may define the resume action for its tool.

Adapters create resume actions only when `/api/resume` asks for one. Discovery and ranking must not leak raw execution details.

## Match policy

The system should distinguish three match types:

### Explicit

User manually linked the current session to the current work object on the current machine.

- Always rank first.
- Confidence `1.0`.
- Persisted as `LocalSessionBinding`.

### Inferred

System found evidence in local history.

- Rank below explicit.
- Show evidence before executing.
- Ephemeral resolve output only, unless the user promotes it.

### Suggested

Weak heuristic match.

- Never auto-execute.
- Present as “possible match”.
- Ephemeral resolve output only, unless the user promotes it.

## UX states

### No match

- Show “Link current session”
- Show optional nearby candidates if confidence clears a low threshold

### One strong match

- Show primary `Resume`
- Show `Copy command`
- Show why it matched
- Show `Exec` only if local exec is enabled and the match is explicit or user-confirmed

### Multiple plausible matches

- Show ranked list
- Never auto-resume
- Let user pick and optionally save explicit link

## Security defaults

### Default release posture

- copy-first
- explicit opt-in for local execution
- local exec is configurable and off by default
- connector never sends raw commands
- companion validates action type against adapter output
- localhost HTTP only in v1, hardened with loopback binding, extension-origin allowlist, and a per-install token
- token setup uses an explicit pairing flow; the extension cannot read local files directly
- exec actions must launch a visible terminal/editor surface, never a headless background CLI process

### Prototype-only behavior to remove

- hard-coded paths
- wildcard cross-origin assumptions
- dangerous bypass commands enabled by default
- implicit iTerm availability as exec permission

## Suggested companion API

### `GET /health`

Returns availability and capability flags.

### `POST /api/resolve`

Input:

```json
{
  "protocol_version": 1,
  "url": "https://www.drupal.org/project/canvas/issues/3558241",
  "page_title": "Issue title if available",
  "metadata": {
    "canonical_url": "https://www.drupal.org/project/canvas/issues/3558241",
    "issue_key": "3558241"
  }
}
```

Output:

```json
{
  "protocol_version": 1,
  "work_object": {},
  "matches": [
    {
      "candidate_ref": "resolve-opaque-candidate-ref",
      "agent": "codex",
      "resume_label": "Codex session from Apr 8",
      "workspace_path": "/Users/scott/dev/drupal-contrib/projects/canvas",
      "summary": "Reviewer follow-up on unresolved MR comments.",
      "match_type": "inferred",
      "confidence": 0.82,
      "evidence": []
    }
  ],
  "capabilities": {
    "can_exec": false
  }
}
```

Error responses must use structured codes:

```json
{
  "protocol_version": 1,
  "error": {
    "code": "unsupported_url",
    "message": "No Direct Resume connector supports this page."
  }
}
```

### `POST /api/link`

Creates or updates a portable work anchor and a machine-local session binding.

Valid inputs:

- `candidate_ref` from the latest resolve result
- or manual `{ agent, session_id, workspace_path }`

The resulting local binding is explicit authoritative state.

### `POST /api/resume`

Requests a resume action for a chosen candidate.

Should support:

- `mode: "copy"`
- `mode: "exec"`

`mode: "exec"` is valid only when local exec is enabled and the selected binding is explicit or user-confirmed.

`mode: "exec"` must launch into a visible configured launcher, for example Terminal.app, iTerm, or an editor URI. If no launcher is configured, the companion should return a copy-safe fallback.

Input should use an opaque candidate or binding reference, not a raw shell command.

Output should be a typed action object:

```json
{
  "protocol_version": 1,
  "action": {
    "type": "copy",
    "label": "Copy Codex resume command",
    "command": "codex resume 019c0cd9-4367-77d0-a0a2-a5ebbb15ab27",
    "workspace_path": "/Users/scott/dev/drupal-contrib/projects/canvas"
  }
}
```

Before returning any action, the companion must re-run liveness and policy checks.

### `POST /api/setup/pair`

Pairs the extension with the companion.

The exact UX can be simple in v1:

1. User runs `direct-resume setup`.
2. CLI prints a short-lived, single-use pairing token or opens a local setup page.
3. User pastes/confirms the token in the extension.
4. Extension submits the token in a header or request body.
5. Extension stores the long-lived companion token in extension storage.
6. Companion stores the matching token under `~/.direct-resume/`.

Do not assume the extension can read the token from local disk.
Do not put pairing tokens in query strings.

## MVP recommendation

Start explicit-link first.

Flow:

1. User opens issue page.
2. Extension resolves work object.
3. If no explicit link exists, user clicks `Link current session`.
4. User selects a detected candidate or pastes a session id and workspace path.
5. Companion stores a portable work anchor plus a local session binding.
6. Later, the issue page shows `Resume`.

Auto-detect should come second, once there is enough linked data to rank against and evaluate.

## Test requirements

- Unit tests for Drupal and Jira canonicalization.
- Unit tests for portable anchors versus local session bindings.
- Companion HTTP integration tests for resolve, link, resume, origin/token rejection, and exec gating.
- Small DOM/message harness for extension no-match, one-match, multi-match, offline, copy-only, and exec-enabled states.
- One thin seam-level smoke test crossing extension message handling, localhost auth, resolve, and resume using fake connector/adapter implementations.
- Regression coverage for the existing Drupal issue flow after moving from issue-id routing to WorkObject routing.
- Tests for stale local bindings where the adapter reports `isAlive(candidate) === false`.
- Tests for resolve metadata hints, including canonical URL and Jira issue key extraction.
- Tests proving exec falls back to copy when no visible launcher is configured.
- Tests proving bare numeric aliases score lower than URL or connector-key matches.
- Tests proving inferred/suggested matches are not persisted as local session bindings until user confirmation.
- Tests proving `/api/resume` uses opaque references and re-runs liveness/policy checks.

## Performance policy

- Explicit local bindings resolve before history scanning.
- Explicit local bindings must pass adapter liveness checks before display.
- Session adapters may cache discovery results by source file path, mtime, and size.
- Beads should not store the full local session index.
